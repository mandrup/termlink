# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run dev` — run the app directly from TypeScript source via `tsx` (fastest loop for manual testing; this is a terminal UI, so it needs a real TTY).
- `npm run build` — type-checks with `tsc --noEmit`, then bundles `src/index.tsx` with esbuild into a single executable `dist/index.js`.
- `npm start` — runs the built `dist/index.js` (what the published `termlink` binary does).
- `npm test` — runs the full suite once with vitest.
- Single test file: `npx vitest run tests/lib/ssh/ssh.test.ts`
- Single test by name: `npx vitest run -t "resolves with exit code 0"`
- `npm run format` — formats the whole repo with Prettier (single quotes, no semicolons, 100-char print width — see `.prettierrc.json`).

## Style conventions

- Prettier enforces single quotes, no semicolons, and a 100-char print width; run `npm run format` rather than hand-formatting.
- No comments in source or test code. Identifiers should be named clearly enough that the code doesn't need them.
- Import via the `@/*` path alias (maps to `src/*`), not relative `./`/`../` paths — e.g. `import { runInteractive } from '@/lib/pty'`. This is wired into `tsconfig.json` paths, `vitest.config.ts` resolve.alias, and esbuild resolves it natively at bundle time.
- No `.js` extensions on `@/...` imports. This only works because the production build bundles everything with esbuild; `tsc` never emits JS (`noEmit: true` in `tsconfig.json`) so there's no separate compiled-output resolution step to satisfy.
- A directory can stand in for a single-file module: `@/components/ConnectionForm` resolves to `src/components/ConnectionForm/index.tsx` under `tsc`'s `Bundler` resolution, esbuild, and Vitest alike, so a component with sub-components doesn't need its external import path to change when it grows a folder.
- Tests live under top-level `tests/`, mirroring `src/`'s structure (e.g. `src/lib/ssh/ssh.ts` <-> `tests/lib/ssh/ssh.test.ts`), not alongside the source files.
- `vitest.config.ts` pins `FORCE_COLOR: '0'` for the test process. Ink/chalk's color-support detection otherwise depends on the invoking environment (a plain terminal vs. an editor's test runner), which can insert ANSI codes between adjacent differently-styled `<Text>` nodes and break `lastFrame()` substring assertions non-deterministically.
- Commit messages follow [Conventional Commits](https://www.conventionalcommits.org/) (`feat:`, `fix:`, `chore(deps):`, etc.); don't add a `Co-Authored-By` trailer.

## Architecture

termlink is an Ink (React for CLIs) TUI for managing SSH/RDP connections, persisted locally. Entry point is `src/index.tsx`, which loads persisted state and renders `src/App.tsx`. `App.tsx` owns top-level state (mode, clone source) directly and delegates the rest to hooks: `src/hooks/useConnectionsStore.ts` (connections, credentials, selection, search, status message), `src/hooks/useConnect.ts` (the connect flow), and `src/hooks/useReachability.ts` (SSH reachability polling). Ink's `useInput` handler in `App.tsx` doesn't dispatch on keystrokes directly — it calls `src/lib/keybindings.ts`'s `resolveKeyActions`, a pure function that turns `(mode, input, key)` into a list of actions, which `App.tsx` then executes; this is what makes keybinding logic unit-testable without rendering Ink. Presentational pieces (`ConnectionList`, `DetailPanel`, `ConnectionForm`, `HelpScreen`, `ConfirmDialog`, `StatusBar`) live in `src/components/` and are largely stateless, driven by props. `ConnectionForm` is a small folder rather than a single file: `index.tsx` orchestrates state and delegates rendering to `FormFieldRow.tsx` and `NotesPane.tsx`, with its pure logic (field visibility, validation, the form's own keybinding resolution) living in `src/lib/connections/connectionForm.ts` rather than the component.

`mode` (`"list" | "add" | "edit" | "search" | "confirmDelete" | "help"`) determines both what's rendered and how `resolveKeyActions` interprets keystrokes — check that function before adding a new keybinding or screen.

`src/lib/ssh/sshConfig.ts` backs the `i` (import from `~/.ssh/config`) keybinding. The one non-obvious decision there: an imported connection's hostname is the config **alias**, never the block's resolved `HostName`. termlink execs the real `ssh`, so passing the alias keeps the user's whole ssh config applying at connect time — importing `HostName` instead would silently drop `ProxyJump` and every other per-host option. `Port` is still copied explicitly because `buildSshArgs` always passes `-p` (which would otherwise mask a config `Port` with the default 22); `User`/`IdentityFile` are copied so they're visible in the UI. Re-imports are safe: `useConnectionsStore.importSshHosts` skips aliases whose name already exists, case-insensitively.

Connections and credentials are separate, associated via `credentialId` (see `src/models/connection.ts`). `CredentialModel` holds a username plus a `hasPassword` flag -- the password itself lives in the OS secret store via `src/lib/keychain.ts` (macOS `security`, Linux `secret-tool`, Windows PowerShell+DPAPI), keyed by credential id, so nothing sensitive is ever written to the config file. The edit form never reads the secret back; it tracks a `passwordTouched` flag to distinguish "leave the stored password alone" from "clear it". Writes go through a temp-file-then-rename in `saveState` for atomicity, persisted at `$XDG_CONFIG_HOME/termlink/connections.json` (or `~/.config/...`).

### Connecting to a host: pty relay, not stdio inheritance

`src/lib/pty.ts`'s `runInteractive` is the trickiest part of the codebase. Connecting used to spawn the child (`ssh`, now also `xfreerdp`) with `stdio: "inherit"`, sharing the real terminal fd directly with the child — this created a real, hard-to-close race where a keystroke typed right as the session started could be silently dropped, because both Ink and the child were reading the same fd. The fix: spawn the child attached to its own pty (via `node-pty`) and manually relay bytes between the real terminal and that pty ourselves, so Node/Ink never hands over exclusive ownership of the real stdin. `runInteractive` must always be called from inside Ink's `useApp().suspendTerminal()` (see `useConnect.ts`'s `connect`), which turns off Ink's own raw-mode input handling first and restores/redraws once the callback resolves.

`node-pty` ships a native addon whose prebuilt `spawn-helper` binary can lose its executable bit on install; `scripts/fix-node-pty-permissions.js` runs as a `postinstall` step to restore it, since without it connecting crashes with `posix_spawnp failed.`.

SSH ships everywhere, so `src/lib/ssh/ssh.ts`'s `buildSshArgs` just builds args for the real `ssh` binary — `-p`, then `-i` if an identity file is set, then the connection's free-form `extraArgs` (split on whitespace via `splitArgs`, no quoting support) for anything not otherwise modeled, e.g. `-J bastion.example.com` for a jump host, before the final `user@host` target. The binary name itself is resolved by the platform-aware `sshCommand()`, not hardcoded to the `SSH` protocol constant: on Windows it resolves to `ssh.exe` rather than bare `ssh`. This matters only for the interactive connect path (`connect.ts`'s `buildLaunch`, which feeds `pty.ts`'s `node-pty` spawn) — node-pty's Windows backend does its own manual `PATH` search for the executable and, unlike a normal Windows `CreateProcess` call, never appends a default `.exe` extension, so a bare `ssh` silently fails to resolve even though `ssh.exe` is on `PATH` (surfacing as `Failed to start ssh: File not found: `). `execSshCommand`/`probeSshConnection` in `sshExec.ts` don't have this problem since they go through Node's own `child_process.execFile`, whose Windows spawn path does the standard extension search, so they're left defaulting to plain `'ssh'`. `extraArgs` sits before the target for the same reason identity file does: `buildSshExecArgs` (used by both the interactive connect path and MCP's `execSshCommand`) appends the remote command after whatever `buildSshArgs` returns, so anything meant as an ssh flag has to come before that. `src/lib/connections/connect.ts`'s `buildLaunch` is what picks between the SSH and RDP launch (command, args, result formatter, cleanup) for a given connection; `useConnect.ts` just calls it and hands the result to `suspendTerminal`.

`src/hooks/useReachability.ts` drives SSH reachability in `ConnectionList`: rather than adding a second dot next to the existing per-row protocol dot (tried first — looked like a stray double-dot next to every SSH row), the same dot doubles up, going from hollow-and-protocol-colored to filled-and-green/red once a probe resolves. It re-probes an SSH connection whenever its `fingerprint()` — the exact fields `buildSshArgs`/`probeSshConnection` use to reach a host: hostname, port, identity file, extra args, and the linked credential's username — differs from the last check, not just the first time its id is seen; this is what makes fixing a typo'd hostname or switching identity files refresh the dot without waiting for a manual `r`. Renaming a connection or editing its notes deliberately does _not_ retrigger a probe, since neither affects reachability. The fingerprint map is rebuilt from scratch on every `connections`/`credentials` change (not merged into), so deleted connections and ones toggled away from SSH drop out on their own — toggling a connection back to SSH later is treated as brand new rather than reusing a stale result. Probing itself goes through a small (`SSH_PROBE_CONCURRENCY`-limited) worker pool, reusing the exact same non-interactive `probeSshConnection` the MCP server's `get_connection_status` tool uses, so it can never hang on a password prompt. `App.tsx`'s `r` keybinding calls the hook's `refresh()` to force a full recheck regardless of fingerprints. RDP's dot never changes — same reasoning as MCP's RDP exclusion, it has no non-interactive probe path.

RDP has no equivalent universal client, so `src/lib/rdp/rdp.ts` doesn't shell out to a specific one: `prepareRdpLaunch` writes a standard `.rdp` file to a temp dir and hands its path to the OS's own file opener (`cmd.exe /c start` on Windows, `open` on macOS, `xdg-open` elsewhere, picked by `buildOpenCommand`) — whatever's registered for `.rdp` files handles it from there. Windows deliberately goes through `start` rather than invoking `mstsc.exe` directly: `mstsc.exe` _is_ the RDP client, so spawning it directly means the pty child stays alive for the whole session, and since `runInteractive`/`suspendTerminal` don't hand the terminal back until that child exits, the TUI stayed suspended (blank) for the entire RDP session on Windows — `start` launches the association and returns immediately, the same as `open`/`xdg-open` do on their platforms. Both protocols' commands still go through the same shared `runInteractive`, which matters even for RDP's openers: it's what lets a prompt from the launched process reach the real terminal, the same way an unrecognized SSH host key would. Because the OS opener hands off and returns almost immediately rather than blocking until the RDP session actually ends, `prepareRdpLaunch`'s `cleanup()` deletes the temp `.rdp` file only after a short delay, as a best-effort guard against deleting it before the handed-off client has read it.

### Build

Production bundling (`scripts/build.mjs`) uses esbuild to produce one `dist/index.js` with a shebang, marking `node-pty` external (native addon, can't be bundled) along with Ink's optional devtools import path (`react-devtools-core`, `*/devtools.js` — only ever reached via a runtime-gated dynamic `import()`, but bundling it inline would make its top-level import eager). A `createRequire` shim is injected via `banner` because some bundled CJS deps (e.g. `signal-exit`) call `require()` for Node builtins in a way esbuild's ESM output can't otherwise satisfy at runtime. Because of this, `react`, `ink`, and `ink-text-input` are `devDependencies` (fully bundled into the shipped file) — only `node-pty` is a real runtime `dependency`. `src/mcp.ts` (below) is built by a second, independent `build()` call in the same script, since it has none of the TUI's native-addon/lazy-import concerns and its dependencies (`@modelcontextprotocol/sdk`, `zod`) are pure JS — also `devDependencies`, fully bundled into `dist/mcp.js`.

### MCP server: opt-in, allow-listed, not the TUI's stdio

`src/mcp.ts` is a separate binary (`termlink-mcp`) exposing a deliberately narrow slice of termlink to MCP clients — nothing is shared with the TUI process at runtime, they just read the same `connections.json` via `loadState()`. Two restrictions shape everything here, both because an agent is a fundamentally different trust level than a human at the keyboard:

- **Opt-in per connection.** `ConnectionModel.mcp` (`{ enabled, allowedCommands }`) is absent by default; `src/lib/mcp/mcpTools.ts`'s `listMcpConnections` only ever returns SSH connections with `mcp.enabled === true`, and RDP is excluded unconditionally — it has no non-interactive path (see below) and is where the app's one stored secret, the Keychain-backed RDP password, lives. Nothing in this file ever reads or returns `credentialId`.
- **Allow-listed commands, not free-form shell access.** `runMcpCommand` rejects anything that doesn't match one of the connection's `allowedCommands` (exact string, or `"*"`-suffixed prefix) via `src/lib/mcp/commandMatch.ts`'s `isCommandAllowed`. Critically, that function _also_ rejects any command containing a shell metacharacter (`; | & \` $ < >`) independent of the allow-list match — because `ssh host <command>`always hands the joined string to the remote login shell, a command like`"systemctl status nginx; rm -rf /"`would otherwise still satisfy the prefix`"systemctl status "`while smuggling a second command through the`;`.

State is reloaded fresh on every tool call rather than cached at server startup, so disabling a connection from the TUI takes effect on the agent's very next call rather than only after the MCP server process restarts.

Every `run_command` outcome — rejected, succeeded, or a non-zero remote exit — is appended to `src/lib/mcp/mcpAuditLog.ts`'s JSON-lines log (`$XDG_CONFIG_HOME/termlink/mcp-audit.log`), so the user can review what an agent tried after the fact. It deliberately never logs the command's stdout/stderr, only that it ran and how it exited — the log exists to answer "what happened", not to become a second place remote output (and whatever secrets it might contain) ends up on disk. Logging failures are swallowed (`safeLog` in `mcpTools.ts`) so a full disk can't block the actual tool call.

A third tool, `get_connection_status`, runs a fixed probe (`ssh -o BatchMode=yes -o ConnectTimeout=5 ... true` via `src/lib/ssh/sshExec.ts`'s `probeSshConnection`) rather than anything agent-supplied, so it needs no `allowedCommands` entry — `findMcpConnection` in `mcpTools.ts` is the single lookup+opt-in check shared by all three tools, with the allow-list check layered on top of it only for `run_command`. Every MCP ssh invocation sets `BatchMode=yes` — it lives in `buildSshExecArgs`, so the probe and `run_command` both get it: with no terminal behind a tool call, ssh would otherwise hang on a password prompt instead of failing fast. Output handed back to the agent is truncated at `SSH_EXEC_MAX_OUTPUT_CHARS` with a visible marker; `execFile`'s `maxBuffer` is set well above that so ordinarily-large outputs hit the graceful truncation path rather than a killed process.

(Tests for anything touching `mcpAuditLog.ts` must override `$XDG_CONFIG_HOME` to a temp dir, the same way `storage.test.ts` does — a test that forgets to mock the `log` parameter otherwise falls through to the real `appendAuditLog` and writes into the developer's actual config directory, which happened once during this feature's own development.)

Execution itself goes through `src/lib/ssh/sshExec.ts`'s `execSshCommand`, not `pty.ts`'s `runInteractive` — that function exists specifically to relay bytes to a real terminal via `suspendTerminal()`, which is meaningless for a tool call with no terminal behind it. `execSshCommand` is a plain `execFile('ssh', [...buildSshArgs(...), command])` with a timeout, reusing the same arg-building as the interactive path.
